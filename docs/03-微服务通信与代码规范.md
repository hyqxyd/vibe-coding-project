# 课堂 Vibe Coding 平台：微服务通信与企业级代码规范

本文档确立了本平台在 **CQRS（命令查询职责分离）多语言混合架构** 下的跨服务通信标准，并参考了**字节跳动（ByteDance）**及行业头部大厂的 Go/Java 高质量工程实践规范，为团队的协同开发提供准则。

## 1. 跨语言通信架构 (Go <-> Java)

控制面（Control Plane，Java/Spring AI）与数据面（Data Plane，Go）的通信，根据业务场景分为**同步调用**与**异步事件**两类：

### 1.1 同步通信：gRPC (基于 Protobuf)
适用于需要即时响应和强类型约束的指令控制。
- **Java -> Go**（指令下发）：
  - 场景：Java 层要求 Go 启动一个学生沙箱（`StartWorkspace`），获取分配的容器 ID 和 WebSocket PTY 地址。
  - 优势：Protobuf 天然跨语言，性能比 HTTP/REST 高出 3-10 倍，契合 Go 的底层系统调用特性。
- **Go -> Java**（状态查询）：
  - 场景：Go 接收到代码流修改请求，需实时向 Java 查询用户的临时权限 Token（`CheckPermission`）。

### 1.2 异步通信：Redis Stream / Kafka
适用于状态流转、日志解耦和重试机制。
- **Go -> Java**（状态回传与事件推送）：
  - 场景：沙箱容器意外 OOM 崩溃，Go 捕获后推送 `WorkspaceCrashedEvent` 到消息队列，Java 消费后更新数据库状态，并触发 `AI 运行调试 Agent`。
  - 场景：终端执行的日志实时抽样写入 Redis Stream，Java 异步消费用于教师端的监控与留痕。
- **Java -> Go**（批量异步任务）：
  - 场景：全班 100 人一键下课，Java 发送 `BatchDestroyWorkspaceEvent`，Go 异步平滑地回收 Docker 资源，防止瞬间并发导致系统卡顿。

## 2. 字节级代码规范：Go 语言

参考字节跳动青训营与内部工程实践（如 CloudWeGo、KiteX），Go 代码强调**简单性、可读性和并发安全**。

### 2.1 命名规范
- **包名**：全小写单词，禁止下划线，不与标准库冲突（如不要用 `math`、`http`）。
- **变量/函数**：
  - 导出的接口、结构体、函数使用 `UpperCamelCase`（大驼峰）。
  - 私有变量、局部变量使用 `lowerCamelCase`（小驼峰）。
  - 作用域越广的变量名需要越长（表意清晰），作用域极小（如 `for` 循环内部）的变量可简写为 `i`, `v`。
- **接口命名**：单个方法的接口一般以 `er` 结尾（如 `Reader`, `Writer`）。

### 2.2 控制流程与错误处理
- **尽早 Return，避免嵌套**：优先处理错误和特殊情况，保证主流程代码（Happy Path）在最外层（线性原理）。
  ```go
  // Bad
  if err == nil {
      doSomething()
  } else {
      return err
  }
  
  // Good
  if err != nil {
      return err
  }
  doSomething()
  ```
- **Error Wrap**：使用 `fmt.Errorf("...: %w", err)` 包装错误，保留原始错误链，方便追踪。禁止直接忽略错误（`_`）。
- **Panic 与 Recover**：业务逻辑中**禁止主动使用 `panic`**。所有 Goroutine 的入口处必须进行 `defer recover()`，防止单一协程崩溃导致整个 Go 进程宕机。

### 2.3 高性能与内存规约
- **Slice/Map 预分配**：在 `make` 初始化时尽可能提供容量（cap）信息，减少扩容时的内存分配开销。
- **并发安全**：尽量通过 `channel` 共享内存，而不是通过共享内存（锁）来通信。如必须使用锁，锁的粒度应尽可能小。

## 3. 行业级代码规范：Java / Spring 语言

参考阿里 Java 开发手册与字节跳动基础架构规范，Java 代码强调**严谨、面向对象与防御性编程**。

### 3.1 命名与规约
- **类名**：`UpperCamelCase`，异常类以 `Exception` 结尾，测试类以 `Test` 结尾。
- **分层命名**：
  - 控制层：`xxxController`
  - 服务层：`xxxService` / `xxxServiceImpl`
  - 数据层：`xxxRepository` / `xxxDAO`
  - 模型对象：请求参数用 `xxxReq` / `xxxDTO`，响应结构用 `xxxResp` / `xxxVO`，数据库实体用 `xxxEntity` 或 `xxxDO`。

### 3.2 异常与控制流
- **防御性编程（防 NPE）**：
  - 尽量使用 `Optional` 包装可能为空的返回值。
  - `equals` 判断时，常量或确定有值的对象放在前面：`"SUCCESS".equals(status)`。
- **异常处理**：
  - 禁止用 `catch (Exception e)` 吞掉异常而不做任何日志打印。
  - 业务异常必须定义全局异常拦截器（`@RestControllerAdvice`）统一包装为标准 JSON 返回。

### 3.3 AI 与并发规约
- **线程池规范**：禁止使用 `Executors` 默认的工厂方法创建线程池（容易导致 OOM），必须通过 `ThreadPoolExecutor` 明确指定核心线程数、最大线程数、队列大小和拒绝策略。
- **Spring AI 契约**：
  - 所有注入给大模型的 `@Tool`（函数），必须提供极度清晰的 JavaDoc 注释（包含参数含义、返回值含义），因为这会被转化为 LLM 的 Prompt。

## 4. 团队协同约束
1. 跨服务接口（gRPC `.proto` 文件）统一放在独立的仓或目录下管理，通过 CI/CD 自动生成 Go 和 Java 代码。
2. 任何跨语言的 API 变更，必须先提交 Schema 变更的 PR，Review 批准后方可修改业务代码。
