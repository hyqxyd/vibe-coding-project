# 课堂 Vibe Coding 平台 AI 协同开发与提交规范

## 1. 目标

本规范用于多人协同开发且引入 AI Coding 辅助时，统一：

- 分支策略
- 提交消息格式
- 合并节奏
- 冲突预防机制
- AI 改动同步机制

## 2. 提交规范

## 2.1 标准格式

采用行业通用的 Conventional Commits，并兼容阿里/字节常见工程实践：

`type(scope): subject`

示例：

- `feat(schema): add patch conflict detection`
- `fix(runtime): handle container warmup timeout`
- `docs(process): add ai collaboration workflow`

## 2.2 type 枚举

- `feat`：新功能
- `fix`：缺陷修复
- `refactor`：重构（无功能变更）
- `perf`：性能优化
- `docs`：文档更新
- `test`：测试相关
- `build`：构建系统或依赖变更
- `ci`：CI/CD 配置变更
- `chore`：杂项维护
- `revert`：回滚提交

## 2.3 约束规则

- `type` 必须小写
- `scope` 必填，建议使用模块名
- `subject` 使用动词原形，长度建议不超过 72 字符
- 不允许使用 `update`, `modify`, `fix bug` 这类无信息量标题
- 每次提交只做一件事，避免混合多个目标

## 2.4 推荐 scope

- `schema`
- `generator`
- `runtime`
- `teacher`
- `student`
- `admin`
- `scoring`
- `showcase`
- `infra`
- `docs`
- `process`

## 3. 分支策略

## 3.1 主干模型

- `main`：生产可发布分支，仅接受通过评审的合并
- `develop`：日常集成分支，所有功能分支先合入此分支
- `feature/*`：功能分支
- `fix/*`：缺陷分支
- `chore/*`：工程维护分支

## 3.2 命名规范

`<type>/<owner>/<ticket-or-topic>`

示例：

- `feature/alice/schema-patch-validator`
- `fix/bob/teacher-score-timeout`
- `chore/charlie/docs-sync`

## 4. AI 协同防冲突机制

## 4.1 改动认领机制

每个开发者在开始前必须在 `/.ai-collab/BRANCH_REGISTRY.md` 记录：

- 分支名
- 改动模块
- 计划改动文件
- 预计合并窗口

## 4.2 AI 改动记忆机制

每个功能分支在提交前必须更新 `/.ai-collab/CHANGELOG.md`：

- 本次 AI 参与生成的文件
- 关键设计决策
- 对外接口变化
- 潜在冲突点

## 4.3 同步节奏

- 每位成员至少每半天执行一次 `rebase develop`
- 每日固定两次“合并窗口”：建议 12:00 与 18:00
- 每次进入合并窗口前，先完成本地自测与冲突预检查

## 5. Pull Request 规范

每个 PR 必须包含：

- 变更背景
- 变更范围
- 风险点
- 回滚方案
- AI 辅助说明

AI 辅助说明固定四段：

- AI 参与范围
- 人工复核点
- 自动化验证结果
- 尚未覆盖风险

## 6. 合并策略

## 6.1 默认策略

- `feature/*` 合并到 `develop` 使用 `squash merge`
- `develop` 合并到 `main` 使用 `merge commit`

## 6.2 阻断条件

出现任一条件禁止合并：

- 提交消息不符合规范
- PR 模板未完整填写
- 核心检查未通过
- 与最新 `develop` 存在冲突未解决
- AI 改动记忆未更新

## 7. 推荐检查清单

提交前执行：

1. 本地 lint / typecheck / test 通过
2. 对照 `BRANCH_REGISTRY` 确认未越界修改
3. 更新 `CHANGELOG` 记录 AI 改动
4. 提交消息符合规范
5. 同步最新 `develop` 并解决冲突

## 8. 与大厂实践对齐点

本规范对齐了阿里/字节常见实践中的关键原则：

- 主干可发布
- 小步快提
- 强制代码评审
- 提交可追溯
- 自动化门禁前置
- 统一工作流模板

## 9. 落地建议

首期先落实三项硬约束：

- 强制 commit message 规范
- 强制 PR 模板
- 强制 AI 改动记忆更新

第二阶段再引入：

- 自动冲突预警机器人
- 代码 Ownership 审批
- 模块级并行开发看板
