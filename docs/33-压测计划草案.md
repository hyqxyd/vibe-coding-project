# 课堂 Vibe Coding 平台压测计划草案

## 1. 文档目标

本草案用于为首期 MVP 提供统一的压测设计，验证平台在课堂场景下是否具备支撑约 100 人同时使用的能力，并为上线前调优提供依据。

## 2. 压测目标

- 验证 100 人课堂规模下的主流程可用性
- 验证高峰期登录、配置、运行、提交链路的稳定性
- 验证 Docker 容器池、队列、日志链路是否存在瓶颈
- 验证自动评分任务与课堂主链路的资源隔离效果

## 3. 压测范围

首期建议纳入以下范围：

1. 学生登录与进入工作台
2. 课堂配置读取
3. 项目创建与 Schema 拉取
4. Patch 提交
5. 运行请求与日志查看
6. 项目提交
7. 教师查看项目列表与详情
8. 自动评分异步投递

暂不作为首期重点压测项：

- 复杂公告高并发访问
- 历史归档库联查
- 跨课程大规模并行课堂

## 4. 压测环境假设

## 4.1 基础假设

- 单次课堂规模：100 人左右
- 峰值并发目标：不低于 100
- 学生统一使用预装技术栈环境
- 运行请求通过 Docker 容器池分配沙箱
- 自动评分使用异步队列，不与课堂主流程同步竞争

## 4.2 环境要求

- 使用接近生产的数据库、缓存、对象存储和日志链路
- 运行容器池数量提前预热
- 压测环境与上线环境的配置差异需单独记录
- 所有监控指标需提前接入 Prometheus / Grafana 或同级方案

## 5. 核心链路拆分

## 5.1 学生链路

| 链路 | 接口或动作 | 目标 |
| --- | --- | --- |
| 进入课堂 | 登录 + 拉取课堂配置 | 验证读链路首屏能力 |
| 创建项目 | `POST /student/projects` | 验证写入与项目初始化 |
| 获取 Schema | `GET /student/projects/{projectId}/schema` | 验证读链路稳定性 |
| 提交 Patch | `POST /student/projects/{projectId}/patches` | 验证配置与对话写链路 |
| 启动运行 | `POST /student/projects/{projectId}/run` | 验证调度与容器池分配 |
| 查看日志 | `GET /student/projects/{projectId}/run-logs` | 验证日志聚合读取 |
| 提交结果 | `POST /student/projects/{projectId}/submit` | 验证最终提交峰值能力 |

## 5.2 教师链路

| 链路 | 接口或动作 | 目标 |
| --- | --- | --- |
| 查看总览 | `GET /api/teacher/courses/{courseId}/classes/{classId}/overview` | 验证聚合读性能 |
| 查看项目列表 | `GET /api/teacher/courses/{courseId}/classes/{classId}/projects` | 验证分页查询能力 |
| 查看项目详情 | `GET /api/teacher/projects/{projectId}` | 验证详情聚合能力 |
| 查看原始对话 | `GET /api/teacher/projects/{projectId}/conversations` | 验证对话读取性能 |

## 5.3 异步链路

| 链路 | 目标 |
| --- | --- |
| 自动评分任务投递 | 验证提交后任务可快速入队 |
| 自动评分任务消费 | 验证队列处理不挤占主链路 |
| 运行日志落盘与聚合 | 验证日志采集不影响接口延迟 |

## 6. 场景设计

## 6.1 场景 A：100 人同时进入课堂

目标：验证课堂开始时的首屏压力。

动作：

- 100 个学生账号在短时间内登录
- 同时拉取课堂配置、主题、问题与个人项目信息

关注指标：

- 登录成功率
- 配置读取延迟
- 数据库连接池占用
- 网关与鉴权服务 CPU

## 6.2 场景 B：100 人创建项目并提交 Patch

目标：验证写链路与 Schema 存储能力。

动作：

- 100 人在进入工作台后创建项目
- 每人连续提交 3 至 5 次 Patch

关注指标：

- 项目创建成功率
- Patch 写入成功率
- 快照与 Patch 表写入耗时
- 失败重试比例

## 6.3 场景 C：30 人同时运行，70 人继续编辑

目标：验证主流程中最重链路的稳定性。

动作：

- 30 人同时点击运行
- 其余 70 人继续进行配置、Patch 和查看日志

关注指标：

- 容器池命中率
- 运行排队时长
- 容器启动耗时
- 日志采集延迟
- 编辑接口是否被运行链路拖慢

## 6.4 场景 D：100 人接近下课时集中提交

目标：验证课堂尾声高峰提交能力。

动作：

- 100 人在 3 至 5 分钟内集中提交项目
- 提交后同步触发自动评分任务入队

关注指标：

- 提交成功率
- 自动评分投递耗时
- 队列堆积深度
- 数据库写入峰值

## 6.5 场景 E：教师查看列表与详情

目标：验证教师端高聚合页面不拖慢课堂主流程。

动作：

- 教师同时刷新总览
- 连续打开多个项目详情
- 查看原始对话和日志摘要

关注指标：

- 列表查询耗时
- 详情接口耗时
- 慢查询数
- 缓存命中率

## 7. 指标与阈值建议

## 7.1 接口级指标

| 指标 | 建议阈值 |
| --- | --- |
| 登录 / 配置读取成功率 | 不低于 99% |
| 项目创建 / Patch / 提交成功率 | 不低于 99% |
| 运行请求成功进入调度队列比例 | 不低于 95% |
| 教师总览 / 列表 / 详情成功率 | 不低于 99% |

## 7.2 延迟级指标

| 指标 | 建议阈值 |
| --- | --- |
| 首屏配置读取 P95 | 3 秒内 |
| 项目创建 P95 | 2 秒内 |
| Patch 提交 P95 | 2 秒内 |
| 项目提交 P95 | 3 秒内 |
| 教师项目列表 P95 | 3 秒内 |
| 教师项目详情 P95 | 5 秒内 |
| 运行请求进入排队或返回受理结果 P95 | 3 秒内 |
| 预热容器命中时启动到可访问 P95 | 15 秒内 |

## 7.3 资源级指标

| 指标 | 建议阈值 |
| --- | --- |
| 应用实例 CPU 持续占用 | 不长期超过 80% |
| 应用实例内存占用 | 不长期超过 85% |
| 数据库连接池使用率 | 不长期打满 |
| 队列积压恢复时间 | 课堂内可回落到稳定区间 |
| 容器池空闲实例数 | 课堂开始与高峰前保持安全余量 |

## 8. 观测与采集项

压测时建议至少采集：

- 接口 QPS、成功率、P95、P99
- 数据库慢查询数量
- Redis / 队列长度
- 容器池已分配数、空闲数、回收时长
- 日志采集延迟
- 自动评分任务积压深度
- 前端页面首屏耗时与错误率

## 9. 风险重点

- 运行链路与编辑链路共用资源导致相互挤压
- 教师详情聚合接口读取对话和事件过多导致变慢
- 提交高峰时自动评分任务挤占数据库写入能力
- 容器预热不足导致课堂内集中冷启动
- 日志采集链路阻塞导致运行接口回包变慢

## 10. 压测执行建议

## 10.1 执行顺序

1. 单接口基线测试
2. 学生读链路组合测试
3. 学生写链路组合测试
4. 运行链路专项测试
5. 教师聚合读链路测试
6. 提交与异步任务混合测试
7. 全链路课堂场景测试

## 10.2 每轮输出

每轮压测结束后建议输出：

- 测试场景说明
- 并发与持续时间
- 成功率与延迟结果
- 资源占用图
- 异常样本与瓶颈结论
- 是否满足上线阈值

## 11. 调优方向建议

- 对课堂配置、主题、问题等高频读数据增加缓存
- 对教师总览与详情聚合接口做预计算或拆分加载
- 对运行链路使用预热容器池和排队反馈机制
- 对自动评分消费并发设置独立上限
- 对日志摘要与明细读取进行冷热分层

## 12. 通过口径建议

压测建议满足以下条件才视为通过：

- 100 人课堂核心读写链路成功率达标
- 30 人以内同时运行时主流程仍保持可用
- 100 人集中提交时无明显雪崩
- 教师端聚合查询不影响学生主链路
- 核心瓶颈已有明确调优结论或缓解方案
