# 成熟工具借鉴与领域空白分析（v0.3 输入）

## 1. 目标

本文件用于回答两个问题：

- 哪些成熟工具机制值得直接借鉴到当前 AI 协同开发体系
- 这个领域真正的空白在哪里，v0.3 应该创新什么

## 2. 成熟工具可借鉴模式

## 2.1 代码协作平台（GitHub / GitLab）

可借鉴：

- 分支保护与状态检查门禁
- CODEOWNERS 自动审批路由
- PR 模板强制化上下文输入
- Required Checks 阻断式合并

映射到本项目：

- 已落地 `.github/workflows/ai-collab-guard.yml`
- 下一步可加 `CODEOWNERS` 与 branch protection 规则模板

## 2.2 CI 编排（GitHub Actions / Jenkins）

可借鉴：

- 阶段化流水线：lint -> test -> policy -> deploy
- 失败快返（fail fast）
- 可复用 job 模板

映射到本项目：

- 现有 `memory_guard` 与 `ai_test_engineer_check` 已具备 policy gate 雏形
- 下一步可拆分 job 并添加 artifact 留存

## 2.3 可观测体系（OpenTelemetry / Grafana）

可借鉴：

- 统一事件模型
- Trace 串联“提交 -> 评审 -> 合并 -> 回滚”
- 指标看板统一口径

映射到本项目：

- `CHANGELOG + TEAM_INTENT + WEEKLY_REVIEW` 已形成文本型追踪
- 下一步应引入结构化事件输出（JSONL）和趋势看板

## 2.4 LLM 开发治理（LangSmith / PromptLayer 类）

可借鉴：

- Prompt / 决策版本化
- 运行轨迹可回放
- 评测集持续回归

映射到本项目：

- `AGENT_MEMORY` 已记录关键决策与不确定项
- 下一步应增加“失败模式库”和“回归样例集”

## 2.5 模型服务运行（vLLM / SGLang / Ollama）

可借鉴：

- 开发与生产分层部署
- 注册中心（tool registry / model registry）
- 成本与吞吐分级策略

映射到本项目：

- 可采用“开发本地轻量 + 生产集中高并发”双环境策略
- 下一步应建立统一 Tool Registry 与 Agent Capability Registry

## 3. 当前领域空白

尽管成熟工具很多，但“多人 + 多 AI + 持续协同记忆”仍有明显空白：

1. **跨 AI 上下文共享标准缺失**  
   不同 AI 工具之间几乎没有统一的协同记忆协议。

2. **AI 改动责任归属不清晰**  
   很多团队只能知道“代码变了”，却很难知道“哪个 AI、基于什么上下文、为何改成这样”。

3. **记忆到门禁的闭环不足**  
   常见实践里记忆是文档，门禁是 CI，二者未形成强绑定。

4. **AI 复核角色工程化不足**  
   多数项目有 AI 生成，但缺少 AI Reviewer 与 AI Test Engineer 的结构化职责。

5. **同步与异步策略混乱**  
   缺少明确“何时异步播报、何时升级同步处理”的协议化机制。

## 4. v0.3 创新方向

## 4.1 协同记忆协议标准化

新增 `AI-COLLAB-MEMORY-SPEC`：

- 统一字段（branch/file/agent/risk/validation）
- 统一事件类型（async_update/sync_merge/review_reject）
- 统一输出格式（markdown + jsonl 双写）

## 4.2 冲突预测引擎

基于历史 `FILE_MEMORY + CHANGELOG` 计算冲突风险分：

- 文件冲突频率
- 同时编辑窗口重叠率
- 跨模块耦合度

输出：

- `risk_score`
- `recommend_merge_order`
- `reviewer_route`

## 4.3 Harness 质量网关

将 AI Builder / Reviewer / Test Engineer 三角色串成自动链路：

- Builder 提交改动与上下文
- Reviewer 进行一致性审查
- Test Engineer 进行正确性与回归审查
- Human Owner 做最终仲裁

## 4.4 团队知识库检索层

在 `TEAM_INTENT` 之上增加查询能力：

- “最近 7 天高风险改动是什么”
- “某文件最近三次改动为何发生”
- “哪些 AI 任务反复被 reject”

## 5. v0.3 两阶段实施

## 阶段 A（1-2 周）

- 落地 memory spec v0.1
- 输出 jsonl 事件流
- 增加 conflict risk 初版计算脚本

## 阶段 B（第 3-4 周）

- 落地 reviewer/test engineer 自动路由
- 增加周度趋势看板输入数据
- 建立失败模式回归用例集

## 6. 验收指标

- 冲突预测命中率
- AI 复核拒绝后缺陷逃逸率下降
- 合并窗口平均阻塞时长下降
- 相同问题重复修复次数下降

## 7. 结论

这个领域确实存在明显空白，尤其在“多 AI 协同记忆标准化 + 质量闭环自动化”上。  
当前 v0.2.1 已经构建了可执行基础，v0.3 的价值在于把这套基础从“机制”升级为“协议 + 预测 + 自动路由”。
