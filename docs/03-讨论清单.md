# 课堂 Vibe Coding 平台讨论清单

## 1. 当前已确认结论

### 1.1 教学组织与任务粒度

- 一次课堂不必只对应一个主题，主题主要用于管理和检索
- 一个主题下允许多个问题，便于教师组织和学生查找
- 学生允许自由发挥超出题目边界，但系统应根据画像和任务目标给出提示与引导

### 1.2 学生自由度边界

- 允许自由输入任何需求，但系统对无法完成的需求必须明确告知原因与替代方案
- 不强制从教师模板开始，模板只是辅助入口
- 首期不建议开放额外依赖安装，统一使用平台预装环境
- 统一基础环境建议内置 LangChain、前端样式库与常用依赖，避免依赖安装导致流程变慢

### 1.3 教师干预深度

- 首期教师以查看、分析、评分为主，不直接下发修复建议
- 首期不提供冻结学生工作区能力
- 首期允许教师在课堂中动态调整资源配额

### 1.4 统一模型与运行底座

- 接受以 Project Schema 作为单一事实源
- 不建议允许绕过 Schema 直接自由改源码
- 首期接受 Docker 容器池方案
- 课堂规模约 100 人时，重点是做好 Web / API 沙箱隔离和资源治理，不必一开始直接上 Kubernetes

### 1.5 高并发与数据策略

- 班级规模目标约为 100 人
- 峰值并发按 100 保底设计，并为异常场景预留余量
- 延迟目标以“用户可接受”为准，重点优化体感流畅性而不是极限低延迟
- 对话与行为日志长期保留，在线主库按 4 个月或学期周期归档迁移
- 教师需要查看全部学生原始对话，用于数据分析、评价与评分

### 1.6 画像、推荐与展示

- 首期既要做分析特征，也要做推荐建议，但学生可以选择不采用
- 首期不做“模板市场”这类商业化产品形态，更适合做教学模板库
- 首期不做多人协作
- 需要支持公告展示优秀作品，并公开作品截图、Agent 配置摘要和最终结果访问入口
- 学生端需要有提交结果按钮，并能查看自己的评分

## 2. 仍需进一步明确的问题

### 2.1 数据与合规

- 当前没有额外学校数据合规要求
- 首期不要求原始对话脱敏、分级授权或导出审批流程
- 公告区默认只公开流水线配置视图与最终运行结果，其他数据不默认公开

### 2.2 技术边界

- 学生项目技术栈首期完全固定统一，后期再考虑白名单切换
- 后期可以补充干预组或限制组能力，用于控制不同学生群体的可用能力边界
- 优秀作品运行结果采用限时访问地址，可按需要延长有效期
- 归档库与在线库之间的联动查询不进入首期范围

### 2.3 教学评估

- 过程评分和结果评分的权重由不同课程教师自行配置
- 教师评分需要结合系统自动评分建议，但自动评分不要求实时返回，建议异步生成
- 不同课程需要支持不同评分 Rubric

## 3. 当前优先级

### P0

- 确认统一 Project Schema 与固定流水线作为首期核心架构
- 确认学生项目统一技术栈与预装依赖边界
- 确认 100 人课堂的容器池容量与资源隔离方案
- 确认教师评分、原始对话查看和优秀作品公告属于 MVP

### P1

- 确认画像标签、推荐策略和认知分析可视化方案
- 确认日志归档和冷存储策略
- 确认限时访问结果页的失效、续期和发布流程
- 确认评分 Rubric 配置方式和自动评分触发时机

### P2

- 评估后续是否开放自定义 Agent 编排
- 评估后续是否增加多人协作能力
