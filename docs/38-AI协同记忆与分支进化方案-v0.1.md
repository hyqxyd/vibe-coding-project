# 课堂 Vibe Coding 平台 AI 协同记忆与分支进化方案 v0.1

## 1. 方案定位

本方案用于解决多人协作 + 多 AI Coding 辅助场景下的四类核心问题：

- 同一模块多人并发导致冲突放大
- AI 各自“看不到”他人近期改动导致重复改造
- 合并后知识丢失，后续 AI 再次犯同类错误
- 同步 / 异步协作节奏不统一导致代码不同步

## 2. 约束与原则

## 2.1 约束

- 不依赖任何非授权泄露源码
- 可借鉴公开工程方法与公开最佳实践
- 首期优先低成本可落地，避免一次性引入过重平台

## 2.2 原则

- 主干稳定优先于分支速度
- 记忆可追溯优先于“感觉记得住”
- 小步同步优先于大批量回灌
- 人工兜底优先于 AI 自证正确

## 3. 总体设计

采用“三层记忆 + 双通道同步 + 门禁合并”的机制。

## 3.1 三层记忆

1. **分支记忆**：每个分支改动目标、边界、风险、状态  
2. **文件记忆**：每个关键文件最近改动意图、外部影响、冲突风险  
3. **AI 会话记忆**：各 AI 辅助会话的关键决策、失败经验、待确认项  

## 3.2 双通道同步

- **同步通道（Sync）**：合并窗口内的强一致流程
- **异步通道（Async）**：开发过程中低摩擦增量广播

## 3.3 门禁合并

PR 合并前必须满足：

- 提交规范通过
- 分支记忆完整
- 文件记忆已更新
- AI 风险项有人类复核
- rebase 后冲突检查通过

## 4. 记忆模型

## 4.1 分支记忆模型

每个工作分支维护一张 Branch Card：

- branch
- owner
- touched_modules
- touched_files
- api_or_schema_changes
- ai_generated_ratio
- merge_risk_level
- merge_window
- blockers

## 4.2 文件记忆模型

针对核心文件维护 File Memory：

- file_path
- last_change_branch
- last_change_intent
- interface_impact
- downstream_impact
- conflict_hotspot
- review_required_by

## 4.3 AI 会话记忆模型

每次 AI 辅助任务记录：

- agent_id
- task_goal
- input_context_scope
- generated_files
- key_decisions
- known_uncertainties
- human_validation_result

## 5. 协同流程

## 5.1 开发前

1. 认领分支并登记 Branch Registry
2. 声明改动边界与文件范围
3. 拉取最新 develop 并做冲突预检查

## 5.2 开发中（异步）

1. 每次关键改动后更新 AI 会话记忆
2. 若涉及接口 / Schema，必须更新文件记忆
3. 通过异步播报写入 CHANGELOG

## 5.3 合并前（同步）

1. 进入合并窗口
2. rebase develop
3. 运行 lint / typecheck / test
4. 执行 PR 模板检查
5. 执行 commit message 与门禁校验

## 5.4 合并后

1. 生成“本次合并摘要记忆”
2. 回写到团队共享记忆索引
3. 推送给后续 AI 会话作为默认上下文

## 6. 同步与异步机制

## 6.1 异步机制

触发时机：

- 新增接口
- 改动公共类型
- 修改高风险核心文件
- 引入跨模块依赖

异步产物：

- 一条 CHANGELOG 记录
- 一条 File Memory 更新

## 6.2 同步机制

触发时机：

- 合并窗口
- 发布前冻结窗口
- 高风险冲突升级

同步产物：

- 冲突解决记录
- 合并决策摘要
- 复盘事项

## 7. 多 AI 辅助协同策略

为避免不同 AI 工具产生“上下文孤岛”，建议统一接入同一套协同记忆目录：

- `/.ai-collab/BRANCH_REGISTRY.md`
- `/.ai-collab/CHANGELOG.md`
- `/.ai-collab/FILE_MEMORY.md`
- `/.ai-collab/AGENT_MEMORY.md`
- `/.ai-collab/SYNC_ASYNC_PROTOCOL.md`

并要求所有 AI 在提交前读取以上文件。

## 8. 风险与控制

## 8.1 典型风险

- 记忆文档不更新变成摆设
- AI 自动改动过大导致评审失焦
- 异步播报过多导致噪音

## 8.2 控制措施

- 通过 PR 模板强制检查记忆更新
- 限制单 PR 改动规模
- 对核心模块引入 Owner 审批
- 每周进行一次协同流程复盘

## 9. 试运行计划（两周）

## 第 1 周

- 启用提交规范与 hooks
- 启用分支记忆与 AI 会话记忆
- 固定每日两次合并窗口

## 第 2 周

- 引入文件记忆
- 统计冲突率、回滚率、平均合并时长
- 输出首轮流程优化建议

## 10. 成功指标

- 冲突率下降 30%+
- 因上下文缺失导致的返工下降 40%+
- PR 首次通过率提升
- 合并窗口平均阻塞时长下降
