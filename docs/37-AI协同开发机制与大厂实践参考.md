# AI 协同开发机制与大厂实践参考

## 1. 适用场景

本机制适用于：

- 多人并行开发
- AI Coding 深度参与
- 需求变化快且需要高频合并

## 2. 大厂常见做法

## 2.1 阿里 / 字节共性

- 主干分支保持可发布
- Feature 分支小步快提
- 强制 Code Review
- 强制自动化门禁
- 高风险模块建立 Owner 审批
- 通过统一模板降低协作认知成本

## 2.2 AI 时代新增控制点

- 每次 PR 必须披露 AI 参与范围
- 每次 PR 必须标注人工复核点
- 每日做“AI 改动摘要同步”
- 对外接口变更必须有契约同步记录

## 3. 推荐协同流程

1. 认领分支与模块边界
2. 本地开发 + AI 辅助
3. 更新 AI 变更记忆
4. 同步 develop 并处理冲突
5. 提交 PR + 模板化评审
6. 合并窗口统一入库
7. 发布前统一回归

## 4. 冲突治理模型

冲突等级建议：

- L1：同文件不同区域
- L2：同文件同函数
- L3：同接口契约
- L4：同核心模块架构冲突

处理策略：

- L1 自动合并后人工复查
- L2 必须二人结对处理
- L3 必须同步接口契约文档
- L4 触发技术评审会议

## 5. AI 记忆同步机制

每个分支在日内必须至少两次同步：

- 上午合并窗口前
- 下午合并窗口前

同步内容：

- 目标变更
- AI 生成或重写文件
- 接口/Schema 变更
- 测试结果
- 风险说明

## 6. 关于“源码泄露”可借鉴点

行业中一些公开事件能提供方法论借鉴，但不建议基于非授权泄露内容构建流程。  
建议仅借鉴公开合规材料中的工程原则：

- 明确的分层职责
- 严格的评审门禁
- 自动化校验优先
- 变更留痕与可回放
